git的学习笔记

2017.10.18 JianCheng Tan  热度 ℃

Git 的一种好玩的解释

首先我们来看几个术语
HEAD
这是当前分支版本顶端的别名，也就是在当前分支你最近的一个提交

Index
index也被称为staging area，是指一整套即将被下一个提交的文件集合。他也是将成为HEAD的父亲的那个commit

Working Copy
working copy代表你正在工作的那个文件集

Flow
当你第一次checkout一个分支，HEAD就指向当前分支的最近一个commit。在HEAD中的文件集（实际上他们从技术上不是文件，他们是blobs（一团），但是为了讨论的方便我们就简化认为他们就是一些文件）和在index中的文件集是相同的，在working copy的文件集和HEAD,INDEX中的文件集是完全相同的。所有三者(HEAD,INDEX(STAGING),WORKING COPY)都是相同的状态，GIT很happy。

当你对一个文件执行一次修改，Git感知到了这个修改，并且说：“嘿，文件已经变更了！你的working copy不再和index,head相同！”，随后GIT标记这个文件是修改过的。

然后，当你执行一个git add,它就stages the file in the index，并且GIT说：“嘿，OK，现在你的working copy和index区是相同的，但是他们和HEAD区是不同的！”

当你执行一个git commit,GIT就创建一个新的commit，随后HEAD就指向这个新的commit，而index,working copy的状态和HEAD就又完全匹配相同了，GIT又一次HAPPY了。

注意：这个跟命令 reset 只有一定关系的。当我们使用此命令不同的参数的时候，上面的三个术语有一定的联系。当使用 --hard 的时候，HEAD，INDEX 和 Working copy 都会充值；使用 --soft 的时候，只有 HEAD 改变；使用 --mixed 的时候，只有 working copy 不变！！！

Git 关于 pull，commint，push，merge 的总结

在解释着三个命令行之前，我先说一下：工作目录，缓存区，本地仓库，远程仓库。我们平时在修改代码的时候，被修改的文件都在缓存区。那这时候就要把缓存区的修改文件都提交到本地仓库。如果想提交到远程的服务器，那么我们就提交文件到远程仓库。最后一个是工作目录。这个是你当前修改文件的目录。文件到远程仓库的流程是： 工作区 -> add到缓冲区 -> commit到本地仓库 -> push到远程仓库。
所以这三个命令有了解释：
add：从工作目录拉去本地仓库
pull：从远程仓库拉去文件到本地
commit：从缓存区提交到本地仓库
push：从本地提交到远程仓库
merge：用于合并指定分支到当前分支，主要针对处于一个区域（缓存，本地或远程）的两个分支进行合并的。例如说，你在缓存有两个分支。你在其中一个叫 dev 的分支进行了修改，而 master 分支还没有更改，所以我们要 merge 这两个分支，这样两个分支的内容相同了。

Git 隐藏 stash 的使用

stash 这个命令行，主要是应用于情景在一些工作做到了一半，不得不去做别的事情的时候用的。例如说，你正在写一个新功能的实现代码，突然领导叫你去修改新的BUG。但是你现在写到一半的代码又不想提交，那怎么办？使用命令 git stash 把当前的工作隐藏起来，也就是把当前工作存到了堆栈上。然后你就可以切换到别的分支进行修改 BUG。当 BUG 修改完毕后，使用命令 git stash list 命令查看已存在更改的列表。使用命令 `git stash pop 即可从堆栈中恢复你新功能的代码到当前的工作目录。

Git 关于 reset 命令及其参数 –hard|soft|mixed

reset 的应用场景

reset 一般用来回退版本的。使用场景有：

  1. 把缓冲区的文件退回到工作目录。执行代码：git reset
  2. 回滚最近一次提交。执行代码：git commit -a -m "备注" ->

–hard | soft | mixed

–soft 、–mixed以及–hard是三个恢复等级。使用–soft就仅仅将头指针恢复，已经add的缓存以及工作空间的所有东西都不变。如果使用–mixed，就将头恢复掉，已经add的缓存也会丢失掉，工作空间的代码什么的是不变的。如果使用–hard，那么一切就全都恢复了，头变，aad的缓存消失，代码什么的也恢复到以前状态。

HEAD detached at (detached HEAD 状态)

莫名遇到了一个错误，git 告诉我我当前 HEAD 指向了 e29b3fe,然后我就有点纳闷！查过网上后发现原来是 HEAD 现在指向了没有分支名字的修订版本，也就是说处于游离状态（detachhed HEAD）。当执行命令 git status 的时候，发现了 HEAD detached at e29b3fe 这句话，也就是说指针指向了一个无名指针。也就说，我们做的工作很有可能在这个无名指针上面。所以解决方案是：1.首先，给这个无名分支创造一个临时的分支，执行命令为 git branch tmp e29b3fe(tmp 是分支的名字)。2.然后 merge 到当前分支。执行命令为 git checkout master 和 git merge tmp。3. 最后删除 tmp 的分支，执行命令 git branch -d tmp。
